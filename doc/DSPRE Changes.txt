üß† initspriteresource
This initializes the sprite resource system for the current animation. Think of it as clearing the stage and prepping the actors before the scene begins.

Purpose: Sets up memory and internal structures so that loadspritresource and loadspritmaybe can function properly.

Required: Always call this before loading any sprite layers.

üì¶ loadspritresource X
This loads a sprite resource from the battle system‚Äôs internal pool.

X: The resource index (0‚Äì3 typically). Each index corresponds to a layer or a sprite group.

Example: loadspritresource 0 loads the first sprite layer (usually the attacker or defender).

üé≠ loadspritmaybe A, B, C, D
This is where things get interesting‚Äîit maps sprite resources to specific roles in the animation.

Let‚Äôs decode loadspritmaybe 4, 0, 0, 0:

Param	Meaning
4	Sprite slot ID (used in resetsprite later)
0	Resource index (from loadspritresource)
0	Battle slot (0 = attacker, 1 = target, 2 = partner, etc.)
0	Layer index (used for rendering order)
So this line says: ‚û°Ô∏è ‚ÄúLoad sprite slot 4 using resource 0, mapped to the attacker, and render it on layer 0.‚Äù

üîÅ Typical Usage Pattern
ruby
initspriteresource
loadspritresource 0
loadspritresource 1
loadspritresource 2
loadspritresource 3

loadspritmaybe 4, 0, 0, 0   // Attacker
loadspritmaybe 5, 0, 1, 1   // Target
loadspritmaybe 6, 0, 2, 2   // Partner
loadspritmaybe 7, 0, 3, 3   // Foe
This sets up four sprite slots (4‚Äì7), each tied to a battle participant and a rendering layer. These slots are then used by the particle system to anchor effects, mask sprites, or overlay visuals.

üßº resetsprite X
At the end of the script, this cleans up sprite slot X. It‚Äôs like striking the set after the scene ends.

üß© How This Connects to .spa Files
Your .spa emitters often reference anchors like 0, 1, 10, 25, etc.

These anchors are mapped to sprite slots and screen positions.

If the sprite slots aren‚Äôt initialized correctly, the emitter might spawn off-screen or fail to attach.

‚úÖ Are These Commands Always Used?
Short answer: Yes‚Äîfor most battle animation scripts. But let‚Äôs break it down by purpose and context:

üß± Core Commands You‚Äôll Almost Always Use
Command	Purpose
initspriteresource	Initializes sprite system for the animation. Required before anything.
loadspritresource X	Loads sprite layers (usually attacker, target, etc.). Needed for visuals.
loadspritmaybe A,B,C,D	Maps loaded resources to battle roles and rendering layers. Essential for anchoring particles.
These are foundational‚Äîthey set the stage for your .spa particles to attach to something meaningful (like Quilava‚Äôs sprite or the center of the screen).

üß† Why They Matter
Without initspriteresource: The engine won‚Äôt allocate memory for sprite layers, and loadspritresource will fail silently.

Without loadspritresource: You won‚Äôt have any visual layers to attach particles to.

Without loadspritmaybe: Your particles won‚Äôt know where to spawn‚Äîso they‚Äôll either default to anchor 25 (center screen) or not show at all.

üß™ Exceptions & Edge Cases
Some scripts (like sound-only effects or screen flashes) might skip sprite loading entirely.

Field effects or weather animations sometimes use global anchors and don‚Äôt need battle sprites.

Cutscene-style scripts may use custom anchors or preloaded assets.

But for move animations, especially ones tied to .spa emitters, these commands are practically mandatory.

üß∞ Pro Tip for Custom Scripts
If you're building a new animation from scratch, start with this safe template:

This gives you full coverage for all battle participants and lets you anchor particles to any of them.
